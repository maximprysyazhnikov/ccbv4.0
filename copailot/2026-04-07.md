## Update — 2026-04-09

### Moderate-aggressive scalping profile enabled

After reviewing live `autopost` logs, we found that scalping was being blocked too often by a combination of:
- `quality_gate_pct=80`

This produced frequent outcomes like:

- `UNSTABLE ... pass 1/2`
- `Stoch weak SHORT`
- `Vol ratio < hard min`

### What was relaxed

Updated runtime settings toward a moderate-aggressive profile while keeping core safety filters in place.

Changed in `.env` and `.env.example`:
- `AUTOPOST_STABLE_CHECKS: 2 -> 1`
- `SCALP_VOL_RATIO_HARD_MIN: 0.70 -> 0.60`
- `SCALP_STOCH_SHORT_MIN_GAP: 3.0 -> 2.0`

Changed in database for active user `1126438536`:
- `quality_gate_pct: 80.0 -> 75.0`

### Expected effect

Expected operational effect:
- more scalping candidates will pass on the first valid run
- fewer candidates will be rejected only because of marginal stochastic weakness
- fewer trades will be blocked by slightly-below-average volume
- effective `SHORT` gate becomes `80%` instead of `85%`

### What was intentionally kept strict

These filters were not relaxed on purpose:
- `RSI oversold SHORT`
- `late SHORT near/below S2`
- late-entry Bollinger protections

Reason:
- they protect against chasing already exhausted moves
- relaxing them too early would likely increase overtrading and lower average entry quality

### Fix 1 — consecutive SL is now a cooldown, not a permanent trap

Changed `services/autopost/core.py`.

Before:
- if the last `SYMBOL_MAX_CONSECUTIVE_SL=3` closed trades for a symbol were SL, the symbol could be blocked indefinitely
- because no new trade could open, the losing streak could never be broken

Now:
- `3 consecutive SL` becomes a timed cooldown
- after cooldown expires, the symbol can try again if it passes normal filters
- the crypto symbol list is not reduced

Added env settings:
- `SYMBOL_CONSECUTIVE_SL_COOLDOWN_MIN=360`
- `SYMBOL_BAN_COOLDOWN_MIN=720`

Operational meaning:
- after 3 SL in a row, a symbol rests for 6 hours
- severe symbol-ban condition rests for 12 hours
- after rest, the normal gate/risk pipeline decides again

### Fix 2 — near-miss gate candidates are captured as paper signals

Changed `services/autopost/core.py`.

Goal:
- do not open weak crypto trades
- still collect evidence for candidates that almost passed gate
- use forward paper validation before relaxing thresholds

Added near-miss capture:
- source: `autopost_near_miss_gate`
- destination: `paper_signals`
- decision log marker: `PAPER_NEAR_MISS`

Added env settings:
- `NEAR_MISS_PAPER_ENABLED=true`
- `NEAR_MISS_GATE_GAP_POINTS=1.0`
- `NEAR_MISS_ALLOW_HARD_BLOCKERS=true`

Current behavior:
- if a candidate misses effective gate by up to `1.0` gate point, it is recorded as paper
- it is not added to `trades`
- it does not open a crypto order
- it lets us measure whether `11/14` candidates near an `80%` gate would have worked

Near-miss closed performance:

```sql
SELECT symbol, direction, COUNT(*) AS n,
       SUM(CASE WHEN rr_realized > 0 THEN 1 ELSE 0 END) AS wins,
       ROUND(AVG(rr_realized), 3) AS avg_r,
       ROUND(SUM(rr_realized), 3) AS sum_r
FROM paper_signals
WHERE source='autopost_near_miss_gate'
  AND status='CLOSED'
GROUP BY symbol, direction
ORDER BY sum_r DESC;
```

Recent autopost decision mix:

```sql
SELECT decision, risk_state, COUNT(*) AS c
FROM (
  SELECT *
  FROM decision_log
  WHERE source IN ('autopost', 'autopost_scan')
  ORDER BY id DESC
  LIMIT 200
)
GROUP BY decision, risk_state
ORDER BY c DESC;
```

### Phase 2 — adaptive gate policy

Planned:
- keep base gate at current levels
- add per-symbol/per-direction modifiers
- raise gate for weak recent EV
- optionally reduce gate only when near-miss paper stream proves positive

Candidate policy:
- if symbol-direction near-miss has at least `20` closed samples and positive `avgR`, allow small gate relief
- if recent closed trades have negative EV, add gate penalty
- never reduce gate when volume is below hard threshold
